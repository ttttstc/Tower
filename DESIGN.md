# Tower Design System

> 本文档描述 **当前已经实现** 的 Tower 产品设计与前端交互基线。  
> 它替代旧版“应县木塔 + Treemap 云图 + React 架构”的设想稿，后续所有首页、大盘、详情、编辑、导入相关设计与实现都以本文为准。

---

## 1. 产品上下文

- **产品定位**：本地优先的项目事项管理平台，用一套连续的东方卷轴 / 书架 / 山水视觉，把项目目标、里程碑、事项推进、依赖和导入流程串成一体。
- **主要用户**：项目负责人、产品、设计、研发、测试，以及需要在同一视图中理解项目状态的协作角色。
- **使用方式**：浏览器内运行，无后端依赖；通过 `tower.json`、本地存储和浏览器文件系统能力管理项目数据。
- **当前项目类型**：桌面优先的多页面 Web App，不是营销官网，也不是通用后台模板。

### 路由与页面职责

| 路径 | 页面角色 | 当前视觉母题 |
|------|----------|--------------|
| `index.html` | 项目首页 / 战略总览 | 山水底图 + 左侧纸页 + 右侧层级签 |
| `dashboard.html` | 事项大盘 | 木质书架 + 书脊事项 |
| `detail.html` | 事项详情 | 中央长卷 + 左侧索引卷轴 + 右侧摘要卡 |
| `item-form.html` | 新增 / 编辑事项 | 卷轴编辑台 + 右侧书脊预览 |
| `mockup/import.html` | Excel 导入流程 | 四步卷页向导 |

## 2. 体验目标

### 2.1 设计要传达什么

- **第一眼不是“管理后台”**，而是“有情境感的项目仪表器”。
- **信息不是抽象块**，而是具象物件：目标是纸页卡，事项是书脊，详情是卷轴，导入是仪式化步骤。
- **多页面是连续体验**：从山水首页进入书架大盘，再进入卷轴详情和表单，应该像在同一座建筑里前进。
- **本地优先要有低压力感**：用户打开页面就能看见、改动、保存，不依赖复杂登录或远端协作模型。

### 2.2 不能退化成什么

- 不能变回通用 SaaS 控制台。
- 不能引入默认紫色渐变、玻璃卡、三栏功能宫格这类“AI 模板感”。
- 不能把所有页面重新扁平化成统一白底表单。
- 不能让视觉道具压过信息本身，装饰必须服务理解和导航。

## 3. 总体美术方向

### 3.1 方向定义

- **方向**：东方建筑语汇 + 文书器物系统 + 暖色历史感。
- **装饰等级**：`intentional`，允许贴图、牌匾、角花、卷轴轴头、书脊纹样，但不走过度繁复的重装饰。
- **情绪关键词**：沉稳、工整、可考据、有仪式感、有进度秩序。

### 3.2 视觉母题分工

- **首页**：战略层。用山水与塔景承载“项目全貌”。
- **大盘**：执行层。用书架表示事项集合、层级和依赖。
- **详情 / 编辑 / 导入**：操作层。用纸、卷轴、牌匾承载说明和输入。

### 3.3 共享材质

- **木**：大盘背景、顶牌、深色操作容器。
- **纸**：主内容区、表单区、卡片、索引卷轴。
- **金**：结构边框、标题牌、进度强调、按钮边缘。
- **墨**：正文与标题文本的主色，不使用冷黑。

## 4. 颜色系统

当前页面允许局部色值差异，但必须维持同一暖色家族。

### 4.1 基础色

| Token | 色值 | 用途 |
|------|------|------|
| `tower-ink` | `#2d241a` / `#2b1a0d` | 主文字、深色标题 |
| `tower-wood` | `#26150d` / `#130a06` | 木质深背景、夜色底 |
| `tower-paper` | `#f5eddc` / `#f3e6cb` | 纸面主底 |
| `tower-paper-soft` | `#fbf3e2` | 次级浅纸面、内发光高光 |
| `tower-gold` | `#a47b3d` / `#d3a157` | 边框、装饰、重点标签 |
| `tower-gold-strong` | `#f1cf93` | 顶牌高光、强调描边 |

### 4.2 语义色

| 语义 | 推荐色 | 用途 |
|------|--------|------|
| 完成 | `#5e7440` | 进度环、里程碑、已完成事项 |
| 推进中 | `#b27b2d` / `#ad7a2b` | 主要进行中状态 |
| 延期 | `#bf6f29` | 需要提醒但未完全阻塞 |
| 阻塞 / 风险 | `#aa4b2d` / `#b24d2e` | 风险、阻塞、危险按钮 |
| 未开始 / 冷却 | `#5a4b3a` / `#585552` | 未开始事项、深色层级签 |

### 4.3 用色规则

- 不使用高饱和冷蓝、科技蓝、霓虹紫。
- 绿色只用于“完成 / 稳定推进 / 正向完成度”，避免乱用。
- 红色只给真正的阻塞和风险，不作为装饰强调色。
- 金色是结构色，不是 CTA 主体色的泛滥来源。

## 5. 字体系统

### 5.1 字体选型

| 角色 | 字体 | 用途 |
|------|------|------|
| Display / Heading | `Noto Serif SC` | 页面标题、区块标题、层级卡、卷轴标题 |
| Body / UI | `Noto Sans SC` | 正文、说明、表单、辅助信息 |
| Numeric Accent | `Georgia` | 百分比、关键进度数字 |

### 5.2 排版原则

- 大标题必须有“匾额感”，优先衬线字体。
- 说明文字保持稳定、紧凑、可读，不追求花哨。
- 百分比和进度数值允许单独使用衬线数字来拉开层次。
- 不把英文字体单独做成一套现代无衬线科技体系，英文只是中文界面里的辅助标签。

### 5.3 字号参考

| 层级 | 范围 |
|------|------|
| 页面总标题 | 28px - 36px |
| 一级区块标题 | 17px - 22px |
| 卡片标题 / 项标题 | 14px - 18px |
| 正文 | 12px - 14px |
| 辅助说明 | 10.5px - 12px |
| 关键数值 | 17px - 34px |

## 6. 布局系统

### 6.1 统一画布

所有主页面基于固定设计画布：

- `1672 x 941`
- 页面通过 `transform: scale()` 自适应缩放
- 默认桌面优先，当前不是响应式重排型产品，而是“整幅舞台缩放型产品”

### 6.2 布局原则

- **顶部**：统一使用木质 / 牌匾式导航头。
- **中部主视区**：承载当前页面核心对象。
- **左右侧栏**：承担目录、过滤器、概览卡、帮助信息，不与主视区争抢层级。
- **底部**：用于说明提示、次级动作，不堆大面积功能区。

### 6.3 层级关系

1. 背景场景层：山水 / 书架 / 木墙。
2. 主舞台层：纸页、书脊、卷轴、表单。
3. UI 控制层：按钮、过滤器、进度卡、辅助说明。
4. 浮层层：抽屉、Modal、Tooltip、日报。

## 7. 共享组件语言

### 7.1 顶部牌匾

- 用于页面标题。
- 居中摆放。
- 使用透明背景牌匾素材。
- 文字为大号 `Noto Serif SC`，强对比浅金色。

### 7.2 纸面容器

- 用于首页左侧概览、详情主纸面、导入主纸面、表单纸面。
- 背景必须带轻微纸纹与暖色渐层。
- 使用柔和内描边，不做强投影浮卡。

### 7.3 卷轴容器

- 用于详情页左侧索引、表单左侧索引、导入步骤导航。
- 要保留明显轴头与卷轴边缘，强调“章节导航”。

### 7.4 书脊事项

- 仅用于大盘及相关预览。
- 书脊高度统一，宽度按每层布局密度微调。
- 状态通过书脊底色和底部进度数字传达。
- 锁定 / 未开始事项使用深色书脊。

### 7.5 层级签

- 首页右侧与大盘左侧都使用层级签 / 牌匾体系。
- 每层必须有独立色签，但整体结构一致。
- 激活态允许更亮的边缘光或轻微提亮，不允许夸张发光特效。

### 7.6 按钮

| 类型 | 使用场景 |
|------|----------|
| `primary` | 保存、导入、创建、主要确认 |
| `secondary` | 次级操作、查看更多 |
| `ghost` | 返回、取消、浅色上下文内的保守操作 |
| `danger` | 删除、风险处理、重置 |

按钮必须保持木 / 金 / 纸语汇，禁止改成现代圆润塑料按钮。

## 8. 页面规范

## 8.1 首页 `index.html`

### 目标

- 展示项目名、目标摘要、目标卡、里程碑进度、整体进度和五层入口。
- 作为“进入大盘之前的战略总览页”。

### 结构

- 背景：高清山水塔景图。
- 左侧：纸页信息栏，包含：
  - 项目标题
  - 编辑按钮
  - 项目目标卡片
  - 里程碑进度条列表
  - 范围 / 里程碑说明 / 相关文档折叠区
- 右上：整体进度卡
- 右侧：五层入口卡

### 交互

- 编辑项目按钮打开抽屉，支持修改项目名称、目标摘要、范围、目标卡、里程碑。
- 点击右侧层级卡跳转大盘对应层。
- 点击整体进度卡或中间塔体热区进入大盘。
- 首页数据由 `Store.subscribe()` 驱动实时刷新。

## 8.2 事项大盘 `dashboard.html`

### 目标

- 用书架承载所有事项，提供最快速的全局浏览和钻取入口。

### 结构

- 顶部：返回首页、标题牌匾、整体进度。
- 左侧：楼层牌匾列。
- 中央：五层书架，每层一排书脊事项。
- 右侧：
  - 新增事项按钮
  - 状态图例
  - 责任人筛选
  - 当前数据状态
- 底部：日报、打开 JSON、保存 JSON、导出副本。

### 交互

- Hover 书脊：轻微上浮、显示摘要卡、显示依赖线。
- Click 书脊：进入详情页。
- 责任人筛选：不重排，只做 dim / 高亮。
- `+ 新增事项`：打开事项抽屉。
- 打开 / 保存：通过 `Store.openProjectFile()` / `Store.saveProjectFile()`。
- 今日日报：展示 `Store.buildDailyReport()` 的 Markdown 结果。

## 8.3 事项详情 `detail.html`

### 目标

- 用单页卷轴解释一个事项的上下文，而不是做成普通属性面板。

### 结构

- 左侧：章节索引卷轴。
- 中央：大卷纸主体，包含：
  - 概述
  - 任务拆解
  - 依赖关系
  - 进度日志
  - 风险记录
  - 资源 / 人员
- 右侧：进度环、所属位置、负责人、优先级与复杂度摘要卡。

### 交互

- 左侧索引点击后滚动到对应章节。
- 顶部可跳回大盘或所属层。
- 详情页是阅读页，重心在解释与状态透视。

## 8.4 新增 / 编辑 `item-form.html`

### 目标

- 在保持卷轴语境的同时，承载真正可编辑的事项表单。

### 结构

- 左侧：编辑索引卷轴。
- 中央：表单主纸面。
- 右侧：书脊预览卡与状态摘要。

### 字段范围

- 标题
- 所属层级
- 状态
- 进度
- 工作量
- 负责人
- 风险开关
- 依赖事项
- 备注
- 里程碑标签

### 交互

- 进度与状态保持同步。
- 风险开启后状态转为阻塞语义。
- 右侧书脊预览随表单变化即时更新。
- 支持 `Ctrl+S` 保存。

## 8.5 导入页 `mockup/import.html`

### 目标

- 把 Excel 导入做成可理解、可校验、可回滚的一段完整流程。

### 步骤

1. 上传文件
2. 字段映射
3. 数据预览
4. 导入完成

### 结构

- 左侧：步骤卷轴
- 中央：当前步骤主区
- 右侧：帮助卡与模板说明

### 交互

- 支持 `.xlsx / .xls / .csv`
- 自动字段识别 + 手动校正
- 导入前必须先做预览和校验
- 导入完成后可直接返回大盘查看结果

## 9. 交互与动效规则

### 9.1 动效原则

- 动效只用于解释空间关系和可点击性。
- 所有动效都应该短、稳、克制。
- 不做弹跳、旋转、放大到失真的戏剧化效果。

### 9.2 当前允许的动效

| 场景 | 动作 |
|------|------|
| 书脊 hover | 轻微上浮、亮度提升 |
| 层级卡 hover | 轻微左移 / 提亮 |
| 左侧卷轴导航 hover | 小幅横向位移 |
| 抽屉 / Modal | 240ms 左右滑入或淡入 |
| 折叠区 | 高度展开 / 收起 |

### 9.3 当前禁止的动效

- 全屏滚动视差
- 大面积泛光扫光
- 自动轮播
- 无意义的数字翻牌
- 会影响阅读稳定性的连续浮动动画

## 10. 数据设计与状态同步

### 10.1 共享数据模型

当前项目数据通过 `js/store.js` 管理，核心结构包括：

- `project.name`
- `project.goal`
- `project.acceptanceCriteria`
- `project.goals`
- `project.milestones`
- `project.items`
- `project.owners`

事项项包含：

- `id`
- `title`
- `layer`
- `status`
- `progress`
- `effort`
- `dependencies`
- `assignee`
- `risk`
- `notes`
- `milestones`
- `createdAt`
- `updatedAt`

### 10.2 计算规则

- 整体进度：按事项 `effort` 加权。
- 分层进度：按层内事项 `effort` 加权。
- 里程碑进度：按挂上该里程碑标签的事项集合加权。
- 状态默认进度映射：
  - `done` → 100
  - `in_progress` → 60
  - `overdue` → 40
  - `blocked` → 25
  - `not_started` → 0

### 10.3 持久化

- 首选浏览器 File System Access API 直接打开 / 保存 `tower.json`
- 不支持该能力时回落到下载 JSON
- 本地状态保存在 `localStorage`
- 日快照也保存在 `localStorage`，用于生成今日日报

### 10.4 同步原则

- 首页、大盘、详情、编辑、导入共用同一份 `Store`
- 页面间跳转后重新读取当前共享数据
- 同页内通过 `Store.subscribe()` 触发实时更新

## 11. 技术实现边界

### 11.1 当前实现基线

- 不是 React 应用
- 当前是 **多页面静态 HTML + 原生 ES Module**
- 共享主题在 `tower-theme.css`
- 首页有一套更场景化的局部 CSS，但仍需遵守同一色彩和字体系统

### 11.2 允许做的演进

- 继续打磨现有卷轴 / 书架 / 纸页组件
- 增补可复用的 PNG 贴图资源
- 提升局部交互清晰度与数据可读性
- 扩充导入、日报、里程碑等真实工作流能力

### 11.3 不要做的演进

- 不要重新引回旧版 Treemap 木塔方案
- 不要为了“工程统一”把所有页改成标准后台组件库样式
- 不要混入与当前美术体系冲突的现代科技风 token
- 不要在未确认前把固定舞台式页面改成完全流式响应式布局

## 12. 资源组织规范

| 目录 | 用途 |
|------|------|
| `UX/resource/home` | 首页专用背景与原子素材 |
| `UX/resource/dashboard` | 书架背景、书脊事项、大盘相关贴图 |
| `UX/resource/dashboard/books` | 书脊状态素材 |
| `UX/resource/atoms` | 标题牌匾、角花、卷轴画、灯笼等通用元素 |

规则：

- 新生成或新裁切素材必须保留源文件与可复用成品。
- 命名优先表达用途，不用 `final2`、`new-new` 这类临时名。
- 页面优先复用现有素材，不随意重新画风格不一致的新件。

---

## 13. 设计守则

1. **先保证叙事一致，再补局部华丽。**
2. **任何新页面都要能回答：它属于山水、书架、卷轴、纸页中的哪一类。**
3. **先用暖色和材质建立秩序，再用状态色表达进度。**
4. **表单也要有场景感，但不能牺牲录入效率。**
5. **数据是主角，贴图是语法，不是主角。**
6. **如果一个新组件放进页面后看起来像通用后台模板，那它就是错的。**
